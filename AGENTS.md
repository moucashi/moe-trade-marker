# 项目协作规则

- 当前项目中的接口文档描述、说明、摘要、响应说明和参数说明统一使用中文。
- 处理中文乱码时，优先使用正确文件编码和运行环境配置，不使用 Unicode 码点绕过编码问题；仅 `cmd` 和 PowerShell 脚本允许使用 UTF-8 BOM，其他脚本、源码和配置文件不使用 BOM。确实无法避免时，先说明原因并给出推荐方案。
- 每次修改会影响最终构建产物功能逻辑或玩家可见显示效果的代码后，必须根据更新内容按照三段式语义化版本号规则同步更新版本号；如果需要执行构建，版本号修改必须在构建之前完成。
- 仅修改构建脚本、打包流程、文档、协作规则、测试工程、格式化配置等工程基建，且不改变最终构建产物的功能逻辑和玩家可见显示效果时，不推进版本号。
- 用户要求提交本项目代码时，必须先检查当前版本号相对上次提交是否发生变化；若版本号未变但确实存在有效更改，则在提交前根据更新内容更新三段式语义化版本号，并在提交后打上对应的版本号 tag。
- 提交代码时必须使用 `git-commit` skill，并且提交说明使用中文。
- 用户要求提交代码时，只执行提交相关操作；复核时即使发现疑似问题，也必须先提醒并询问处理方式，得到确认后才能修改已有代码。
- 除非用户明确要求或确认，否则不要启动开发服务器。
- 处理 SPT 客户端 UI、BepInEx、Harmony/HarmonyX patch 时，必须优先基于当前客户端 `Assembly-CSharp.dll` 等实际程序集确认调用链、目标方法是否已实现、参数顺序和运行时类型；不要只凭类名、基类方法或旧经验推断。
- 客户端 UI patch 优先选择非泛型、已实现、贴近业务判断的稳定方法；避免优先 patch 泛型方法、抽象/基类空实现、编译器生成闭包或纯 UI 后处理点。能在原版可用性判断层处理时，不要只在按钮渲染后强行改 UI 状态。
- Harmony patch 面对混淆参数名时优先使用 `__0`、`__1` 等位置参数或 `object[] __args`；只有确认参数名稳定时才依赖原始参数名。
- 客户端 patch 交付前除了构建通过，还必须检查安装后的 `BepInEx/LogOutput.log`，确认插件加载成功、没有 `HarmonyException`、`IL Compile Error` 或客户端初始化失败；一旦 `PatchAll` 失败，必须视为本次客户端 patch 未生效。
- 对风险较高的客户端 UI patch，尽量缩小 patch 范围并避免一个可选交互 patch 的失败导致全部客户端 patch 中断；必要时拆分安装或增加运行时日志来证明目标 patch 已命中。
